### commands.json
This file is generated by scanning the AddInCommands.cs file using:

```powershell
@{ commands = (get-content .\AddInCommands.cs | where { $_ -match 'Task\s(.+?Cmd)\(' } | foreach { $matches[1].substring(0, $matches[1].length - 3)}) } | ConvertTo-Json -Depth 5
```

### manifest.json
This is a control file used to map query names to their Athena IDs and
to bind query results to the correct table on the dashboard page.

Fetch new query IDs using:

```powershell
aws athena list-named-queries --query "NamedQueryIds" | convertfrom-json | % { aws athena get-named-query --named-query-id $_ | convertfrom-json | select @{Name='Name';Expression={$_.NamedQuery.Name}}, @{Name='Id';Expression={$_.NamedQuery.NamedQueryId}} } | where { $_.Name.StartsWith('Report') } | sort Name
```

## telemetry.js

The dashboard loads entirely client-side. `telemetry.js` reads `manifest.json` to discover which Athena named queries to run, fires them all concurrently via the Query Lambda API, and renders the results as sortable HTML tables.

### How it works

1. **`loadManifest()`** — fetches `manifest.json` and `commands.json` in parallel, then calls `loadSubtitle()` and `loadReport()`.

2. **`loadSubtitle()`** — runs the `ReportStartTime` query and renders "Reporting since first event on [date] at [time]" under the page title. If the request is rate limited, displays "Rate limited. Please try again in an hour." in red instead.

3. **`loadReport()`** — iterates all manifest entries that have a `target` element ID and fires `loadSection()` for each one concurrently via `Promise.all`.

4. **`loadSection()`** — runs the query, renders the result table into the target element, and applies optional post-processing (sortable columns, heatmap, event type row colouring, unused commands list).

5. **`runQuery(queryId)`** — POSTs `{ queryId }` to the API Gateway endpoint. The Lambda resolves the named query and runs it in Athena. The response is `ResultSet.Rows` — the same array format as a direct `GetQueryResults` SDK call. A 429 response throws a tagged error (`err.status = 429`) so callers can distinguish rate limiting from other failures.

### API endpoint

```
POST https://uetc84spi9.execute-api.us-east-1.amazonaws.com/prod/query
Content-Type: application/json

{ "queryId": "<named-query-guid>" }
```

No API key or authentication required. Rate limited to 1 request per hour per public IP. The developer's home WAN IP is whitelisted for unrestricted access (managed via SSM Parameter Store in the infrastructure).

### Why fetch instead of the AWS SDK

The original dashboard used the AWS SDK v2 directly in the browser with Cognito unauthenticated credentials embedded in the page source. This had two problems:

- **Cost exposure** — any bot or automated job could trigger Athena queries, each of which scans and bills against the S3 data.
- **Credential exposure** — even unauthenticated Cognito credentials are usable by anyone who reads the page source.

The Query Lambda proxy removes all AWS credentials from the browser. The Lambda handles Athena authentication server-side, and rate limiting at the Lambda level (per IP, via DynamoDB) prevents abuse.

### Rate limiting behaviour

Rate limiting is enforced in the Lambda, not at the API Gateway level. API Gateway method throttling was considered but rejected because it applies globally — it cannot distinguish the developer's IP from a public visitor's, which made testing painful.

When a public IP exceeds 1 request per hour, the Lambda returns HTTP 429 with a JSON body and `Access-Control-Allow-Origin: *` CORS header. The browser receives the response cleanly (no CORS error); `runQuery` throws an error tagged with `err.status = 429`; `loadSubtitle` catches it and shows the red rate-limited message.

### Table rendering

`renderTable(rows)` converts `ResultSet.Rows` to an HTML table. The first row is always the header. Columns named `Pct` are formatted as `x.x%`. Rows whose first cell contains "total" get the `grand-total` CSS class and are pinned to the bottom during sorting.

### Sorting

`makeSortable()` adds click handlers to all column headers except `Pct`. Clicking a header sorts the data rows ascending/descending, with `grand-total` rows always remaining at the bottom. Sort direction is tracked in `data-sort` on the `<th>` element.

### Heatmap

`applyHeatmap(tableEl, colName)` applies a white → light blue → dark blue → red colour gradient to the named column, where the highest value in that column gets the most saturated colour. Used on count columns to make hot spots visually obvious. Colours are brightened 75% toward white so text remains readable.

### Event type colours

`applyEventTypeColors()` colours rows in the event type summary table: error rows get a light red background, event rows get a light green background.

### Unused commands

`renderUnusedCommands()` compares the command names returned by the `ReportCommandCounts` query against the full command list from `commands.json` and renders any commands with zero recorded events into a separate table.
