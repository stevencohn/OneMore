Assessment: Issue #2217 — Layouts vs. Favorites

 

Context

Issue #2217 started as a narrow CLI request (goto should be able to target a specific OneNote window among several open ones, non-destructively). You broadened the issue body into a "Layouts Management" feature: Save Layout / Restore Layout commands, a Manage Layouts dialog, and savelayout/restorelayout CLI verbs. In a follow-up comment you flagged the real open question: a layout is fundamentally "a folder of favorites, perhaps with window coordinates," and you'd "almost considered adding to Favorites menu" but worried it's "technically a separate architectural concern" that "may add confusion to the Favorites menu."

 

This document is analysis only — pros/cons/alternatives for how to build Layouts relative to Favorites — not an implementation plan. It's grounded in the actual Favorites schema/code and a Plan-agent critique of the alternatives below.

 

Ground truth from the codebase

 

Favorites today (OneMore/Commands/Favorites/):

 

  • Favorite (FavoriteModels.cs:38): ID, FolderID (0=root; single-level folders, no nesting), Name, Alias, Location (path string), Uri, NotebookID, SectionID, PageID, SortOrder, runtime Status (Known/Suspect/Unknown).
  • Persisted in SQLite (%AppData%\OneMore\OneMore.db) via FavoritesProvider : DatabaseProvider. Schema (FavoritesDB.sql):
    CREATE UNIQUE INDEX idx_favorite_target_page ON favorite(pageID) WHERE pageID IS NOT NULL;
    CREATE UNIQUE INDEX idx_favorite_target_section ON favorite(sectionID) WHERE pageID IS NULL;
    A page (or section) can be referenced by at most one favorite row in the entire database — globally, not per-folder. This is the single most important constraint for this decision.
  • FavoritesChecker does "auto-path" repair (recent #2230): re-resolves stale IDs by walking the Location path, patching the row in place. It implicitly assumes one row per page.
  • UI (FavoritesDialog, ManageFavoritesDialog/MangeFavoritesControl, FavoritesMenu) is built entirely around one favorite = one click = one navigate-in-current-window. There is no "restore everything in this folder at once" verb anywhere — folders are passive containers that just expand on the ribbon menu.
  • No window-position/geometry fields exist anywhere in the model.

 

Multi-window / goto state:

 

  • OneNote.GetWindows() already enumerates all open OneNote COM windows — handle, process/thread ID, bounds (Native.GetWindowRect), current notebook/section/page, active flag — but it's wired up for a diagnostics command only.
  • GotoCommand (Cli/Commands/GotoCommand.cs:38) just calls onenote.NavigateTo(pageId, objectId) against whatever OneNote considers "current." It has no concept of a target window at all — this is exactly the gap the original issue reporter hit, and it's unsolved today.
  • Native.SetWindowPos exists as a P/Invoke but is unused for repositioning.

 

Repo convention: every named user collection (Favorites, Hashtags, Variables) is SQLite-backed via its own bespoke XProvider : DatabaseProvider and its own .sql DDL. There is no shared generic "named folder of items" abstraction anywhere — each feature rolls its own model from scratch. Snippets/Themes are flat-file, no folders at all.

 

Alternatives

 

Alt A — Extend Favorites (flag a folder as "is a layout"; add nullable window-bounds/index columns to favorite)

  • Pros: Smallest apparent diff. Reuses folder CRUD, ribbon menu plumbing, import/export, auto-path checker.
  • Cons: Collides head-on with idx_favorite_target_page. A layout's whole point is that the same page can legitimately appear in multiple layouts, or in a layout and independently as a plain favorite — the existing global uniqueness constraint forbids both. Relaxing it changes a load-bearing invariant FavoritesChecker depends on, for a feature that doesn't share its purpose (Favorites want de-dupe; Layouts must not). You'd also be bolting a new aggregate "restore all" verb onto a UI model built entirely around per-item actions — a hidden mode where "some folders behave like lists, others trigger multi-window restore."
  • Verdict: smallest apparent effort, highest actual risk. Not recommended.

 

Alt B — Separate parallel feature (new layout/layout_window tables + LayoutsProvider, modeled on Favorites' folder/item split, own ribbon entry, own CLI verbs, no page-uniqueness constraint)

  • Pros: Structurally correct — a layout slot is allowed to repeat a page across layouts or alongside a favorite, by design. Matches the only pattern this codebase actually uses (every collection is its own bespoke provider — this isn't a deviation, it's the house style). Fully isolated: cannot regress Favorites or its checker logic. Each layout-window row carries exactly what a layout slot needs (ordinal, page ref, optional bounds) without contaminating the Favorite model with fields 95% of favorites never use.
  • Cons: More net-new surface area — new schema, provider, dialog, ribbon entry, CLI commands — though each has a direct precedent to copy (Favorites itself, or Hashtags/Variables).
  • Verdict: medium effort, lowest risk. Recommended.

 

Alt C — Shared core schema (kind discriminator: favorite vs. layout, partial unique index scoped to kind='favorite'), separate UX on top

  • Pros: Avoids duplicating the CREATE TABLE/folder-CRUD boilerplate; theoretically lets a layout's pages be queried alongside favorites later.
  • Cons: The two "kinds" don't actually want the same shape — a layout slot needs an ordinal and no alias/Location-repair semantics; a favorite needs alias/Location repair and no ordinal. Force-fitting both into one row means either dead nullable columns on each side or re-deriving kind-specific behavior in code anyway, while paying for conditional constraints (WHERE kind='favorite') that every future schema change to either feature must now reason about. The codebase has zero precedent for a shared generic collection abstraction — this would be the first one, introduced to save roughly one CREATE TABLE statement.
  • Verdict: medium-high effort, medium risk (mostly over-engineering risk, not regression risk). Architecturally elegant on paper, worst fit for this codebase's actual conventions.

 

A 4th axis, not an alternative storage design: scope v1 to skip persisted window geometry

Your stated bar in the issue comments is "non-destructive — open new windows, don't reopen an already-open page," not "restore pixel-perfect window positions." If v1 drops bounds entirely, a layout-window row only needs (layoutID, ordinal, pageID/sectionID, notebookID) — window placement on restore becomes ephemeral runtime behavior (tile new windows reasonably) rather than persisted/replayed coordinates. This shrinks Alt B to a genuinely small schema and makes the A/B/C effort gap almost entirely about the Favorites-coupling question, not field count. Recommend treating persisted geometry as a v2 stretch goal regardless of which alternative you pick.

 

Recommendation

Alt B, scoped without persisted geometry for v1. It's the only option that doesn't touch Favorites' existing uniqueness invariant, fits the codebase's established "one bespoke provider per feature" convention exactly, and keeps the (already substantial) window-orchestration risk fully isolated from any Favorites regression risk.

 

Answering your specific question: how to differentiate visually/conceptually

Don't make folders-of-favorites double as layouts — conceptually a Favorite is "a bookmark to one place," a Layout is "a multi-window session snapshot," and overloading folders adds a hidden mode users won't expect. Concretely:

 

  • Separate ribbon entry point ("Layouts" alongside "Favorites"), separate icon (window/grid glyph vs. star/bookmark), separate verbs (Save Layout / Restore Layout vs. Add Favorite / Goto Favorite).
  • A combined "Manage" dialog with tabs (Favorites | Layouts) is reasonable — both are just CRUD-over-a-named-list UIs, so sharing the management surface reduces dialog sprawl.
  • Keep the restore/navigate ribbon entry points distinct, not merged into one menu — that's where real confusion would happen, because the invocation semantics genuinely differ (single navigate-in-place vs. multi-window orchestration), not because of visual styling.

 

Orthogonal (and likely larger) risk: window targeting itself

Solving "match N desired pages against M open windows, open new windows for the rest, optionally reposition" is real COM/Win32 orchestration work — GotoCommand has no target-window concept today, and OneNote.GetWindows() is wired for diagnostics only. This work is identical under Alt A, B, or C — the storage decision doesn't make it easier or harder. It should be scoped and risk-tracked as its own work item under #2217, and is plausibly the larger of the two risk areas in this issue, independent of the Favorites-vs-Layouts question.

 

Open items you'll still need to decide (not blocking this assessment)

  • Ribbon placement (the issue itself marks this TBD).
  • Whether "Restore Layout" should ever close/repurpose existing windows, or always be strictly additive (your comment leans additive — "open new windows... don't reopen the same page twice").
  • Whether layouts should be importable/exportable like Favorites already are.
